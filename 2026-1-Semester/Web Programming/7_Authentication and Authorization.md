**과정**: 스마트보안 전공 웹 프로그래밍
**범위**: Chapter 7. Authentication and Authorization

## Part 1. Authentication (인증) vs Authorization (인가)
가장 기본이 되면서도 혼동하기 쉬운 두 개념의 명확한 분리에서 출발합니다. 인증이 먼저 이루어져야 신원을 알 수 있으므로 인가가 가능해집니다.

* **Authentication (인증)**
    * **목적**: "당신은 누구인가?(Who are you?)"라는 질문에 답하며, 사용자가 주장하는 신원을 검증하는 과정입니다.
    * **방식 및 결과**: ID/비밀번호, OTP, 생체인식, OAuth 등의 방법이 사용됩니다. 인증에 성공하면 이후의 요청을 위한 **Token(토큰)**이 발급됩니다.
    * **실패 시 오류**: `401 Unauthorized`
* **Authorization (인가)**
    * **목적**: "무엇을 할 수 있는가?(What are you allowed to do?)"에 답하며, 신원이 확인된 사용자가 어떤 자원과 행동에 접근할 수 있는지 권한을 정의합니다.
    * **실패 시 오류**: `403 Forbidden`

> *(이상 교안 1~4페이지 내용 통합 요약)*

## Part 2. ID/PW 기반 인증 흐름
로그인은 단일 작업이 아닌 **식별(Identification)**과 **인증(Authentication)**이라는 두 단계의 프로세스로 구성됩니다.

1.  **식별 (Identification)**: 사용자가 ID(이메일 등)를 제공하여 서버의 데이터베이스에서 자신의 레코드를 찾도록 합니다. ID는 비밀이 아니며, 오직 '누구를 인증할 것인지' 선택하는 역할만 합니다.
2.  **인증 (Authentication)**: 식별된 사용자가 본인임을 증명하기 위해 비밀번호(공유 비밀값)를 제공합니다. 서버는 비밀번호를 평문으로 저장하지 않고 오직 **해시(Hash)**값만 저장해두며, 일치할 경우 토큰을 발행합니다.

**주요 API 흐름**
* **회원가입 (Registration)**: 클라이언트가 `{email, password}`를 전송하면, 서버는 이메일 중복을 확인하고 비밀번호를 bcrypt로 해싱한 뒤 DB에 저장합니다. 평문 비밀번호는 절대 저장되지 않습니다.
* **로그인 (Login)**: 클라이언트가 `{email, password}`를 전송하면, DB에서 이메일로 사용자를 찾고(식별), `bcrypt.compare()`로 비밀번호 일치 여부를 검증한 뒤(인증), JWT 형식의 `access_token`을 반환합니다.

> *(이상 교안 5~9페이지 내용 통합 요약)*

## Part 3. 안전한 비밀번호 저장: bcrypt
비밀번호 저장을 위해 왜 일반 해시 대신 **bcrypt**를 써야 하는지에 대한 원리입니다.

* **해시 함수의 특징**: 임의의 길이를 가진 입력을 고정된 길이의 출력으로 변환하는 단방향(One-way) 함수입니다. 동일한 입력은 항상 동일한 결과(결정론적)를 내며, 입력이 조금만 바뀌어도 결과는 완전히 달라집니다.
* **MD5 / SHA-256의 한계**: 이들은 데이터를 빠르게 처리하도록 설계되었기 때문에 초당 수십억 번의 추측을 하는 무차별 대입 공격이나, 미리 해시값을 계산해두는 레인보우 테이블(Rainbow table) 공격에 취약합니다.
* **bcrypt의 방어 기제**
    1.  **Salt (솔트)**: 비밀번호를 해싱하기 전 무작위 문자열을 추가합니다. 동일한 비밀번호라도 솔트가 다르면 해시값이 달라져 레인보우 테이블을 무력화합니다. 솔트는 해시값과 함께 저장됩니다.
    2.  **Cost Factor (비용 인자/Rounds)**: 해싱 연산을 지정된 횟수만큼 반복하게 하여 고의로 연산 속도를 늦춥니다. 숫자가 클수록 무차별 대입 공격이 어려워지며, 주로 10을 권장합니다.

**예시 구조 파헤치기**
`$2b$10$ws8D6CNTKLuKrGZZpWDo/OmHkc3oC5pl.xU9XN1UkjRfepPYg1nPC`
* `$2b$`: 사용된 알고리즘 (Algorithm)
* `10$`: 비용 인자 (Cost, 2^10번 반복)
* `ws8D6...`: 무작위로 생성된 22문자의 Salt
* `mHkc3...`: 최종 계산된 Hash 값

**작동 원리 (코드 관점)**
회원가입 시 평문은 `bcrypt.hash(plainPassword, ROUNDS)`를 거쳐 해시된 값으로 저장되며, 로그인 시에는 저장된 해시값에 포함된 솔트를 이용해 `bcrypt.compare(plainPassword, hashedPassword)`로 비교하여 참/거짓을 반환합니다.

> *(이상 교안 10~15페이지 내용 통합 요약)*

## Part 4. 무상태성 인증: JWT (JSON Web Token)
서버가 토큰을 직접 저장하지 않아도(Stateless) 서명을 통해 유효성을 검증할 수 있는 표준 기술입니다. 로그인 성공 시 서버가 JWT를 발급하면 클라이언트는 이후 요청마다 이를 첨부합니다.

**JWT의 3단 구조 (점(.)으로 구분)**
이 3가지 요소는 암호화(Encryption)된 것이 아니라 단순히 Base64Url로 인코딩(Encoding)된 것이므로 **누구나 읽을 수 있습니다**. 따라서 민감한 정보는 절대 Payload에 넣으면 안 됩니다.
1.  **Header (헤더)**: 토큰의 타입(`typ: "JWT"`)과 서명 알고리즘(`alg: "HS256"`) 정보를 담습니다.
2.  **Payload (페이로드)**: 사용자를 설명하는 데이터인 **Claims(클레임)**가 들어갑니다. 유저 ID(`sub`), 이메일(`email`), 권한(`role`), 발급일(`iat`), 만료일(`exp`) 등이 포함됩니다.
3.  **Signature (서명)**: 헤더와 페이로드를 Base64로 인코딩한 값을 이어붙인 뒤, 서버만 아는 비밀키(`SERVER_SECRET`)를 이용해 HMAC-SHA256 알고리즘으로 생성합니다. 이 서명 덕분에 데이터 변조를 막을 수 있습니다.

**검증(Verification) 원리**
서버는 전달받은 토큰의 헤더와 페이로드를 자사 서버의 비밀키로 다시 해싱하여, 서명 값과 일치하는지 비교합니다. 값이 다르면(조작됨) 거절하며, 만료 시간(`exp`)이 지나도 거절합니다.

> *(이상 교안 16~24페이지 내용 통합 요약)*

## Part 5. [Lab] Nest.js 인증/인가 구현 핵심
코드를 통째로 외우기보다는 각 계층이 보안을 위해 **어떤 역할(목적)**을 하는지 파악하는 것이 중요합니다.

1.  **AuthModule 설계 방향**
    사용자 조회를 위해 `UsersModule`을 임포트하고, 토큰 생성/검증을 위해 서명 키(Secret)와 만료 시간을 설정해 `JwtModule`을 등록합니다.
2.  **AuthService의 보안 로직**
    * `register()`: DB에 쓰기 전 이메일 중복부터 확인하여 충돌을 막습니다(ConflictException). 평문 비밀번호는 해싱 직후 즉시 폐기됩니다.
    * `login()`: 사용자 식별이나 비밀번호 인증 중 어느 단계에서 실패하든 구체적인 원인 대신 모호한 에러(`UnauthorizedException('Invalid credentials')`)를 던집니다. 이는 공격자가 등록된 이메일을 알아내는 것을(User enumeration) 방지하기 위함입니다. 성공 시에는 인증에 필요한 최소한의 정보만 담긴 JWT를 생성해 반환합니다.
3.  **JwtAuthGuard를 통한 라우트 보호**
    컨트롤러가 아닌 가드(Guard) 레벨에서 토큰 검증을 수행합니다. 요청 헤더(`Authorization: Bearer <token>`)에서 토큰을 추출해 유효성을 검사하고, 통과하면 해독된 페이로드 데이터를 `request.user` 객체에 붙여 컨트롤러로 넘깁니다.
4.  **역할 기반 접근 제어 (RBAC)**
    토큰 검증 이후, 사용자의 역할(USER vs ADMIN)에 따라 접근을 제어합니다. 커스텀 `@Roles(Role.ADMIN)` 데코레이터로 필요 권한을 명시하고, `RolesGuard`가 이를 `req.user.role`과 비교하여 인가 처리를 완료합니다.
5.  **편의성과 깔끔한 코드: `@CurrentUser()` 데코레이터**
    매번 `req` 객체 전체를 주입받아 `req.user`를 꺼내 쓰는 번거로움을 줄이고자 생성합니다. 토큰 페이로드가 담긴 `user` 데이터만 직관적이고 깔끔하게 추출하는 목적을 가집니다.

> *(이상 교안 25~30페이지 내용 통합 요약)*
